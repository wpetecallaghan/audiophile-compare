# End-to-End Testing

E2E tests run against the live staging database using [Playwright](https://playwright.dev).
They verify complete user flows from the browser through the Next.js app and into Supabase ‚Äî
things that unit tests cannot cover.

---

## Core design decisions

### Framework: Playwright

Native Next.js support, `storageState` for session reuse, parallel test execution
with `workers: 1` (staging DB cannot handle concurrent writes safely), and
built-in trace/screenshot capture on failure.

### Authentication: Supabase Admin API

Magic links and Google OAuth both require external services (email inbox, Google UI).
Neither is viable in a headless test runner. Instead, `global-setup.ts` uses the
Supabase Admin API to generate a magic link URL directly, navigates to it in a
headless browser, and saves the resulting session cookies to
`playwright/.auth/user.json`. Every test that needs auth loads from that file ‚Äî
auth happens **once per test run**, not once per test.

Google OAuth is not tested E2E. The unit tests for `OAuthButtons` verify the
correct Supabase API call is made; testing the OAuth dance itself requires
mocking Google's servers, which is out of scope for a staging integration test.

### Test data strategy: [E2E] prefix + dedicated test user

Running against the live staging database means tests cannot wipe tables.
Three rules keep test data safe and identifiable:

1. **Dedicated test user** ‚Äî a fixed `E2E_TEST_USER_EMAIL` account exists in
   staging. All E2E-created data is owned by this user.
2. **Identifiable prefix** ‚Äî all created names begin with `[E2E]`
   (e.g., `[E2E] Lejonklou Sagatun`, `[E2E] Tidal listening test`).
3. **Teardown** ‚Äî `global-teardown.ts` deletes every `[E2E]`-prefixed record
   owned by the test user at the end of every run. Uses the Supabase admin
   client so RLS does not interfere.
4. **Dedicated admin test user (step 64)** ‚Äî a second, permanent fixture
   account, `E2E_ADMIN_USER_EMAIL`, listed in staging's `ADMIN_EMAILS`. A
   separate account from `E2E_TEST_USER_EMAIL`, not the same one made an
   admin ‚Äî the regular test user must stay a non-admin for negative-case
   assertions (e.g. `profile.spec.ts`'s "non-admin doesn't see the Admin
   section") to mean anything. Used only by `admin-clip-override.spec.ts`,
   which runs under its own `admin` Playwright project (see
   `playwright.config.ts`) with its own `storageState`
   (`playwright/.auth/admin.json`), generated by `global-setup.ts`
   alongside the regular session.

Tests that read existing staging data (e.g., the public feed) must be read-only
and must not assert on exact record counts ‚Äî they assert on structure.

## File structure

```
e2e/
  global-setup.ts          ‚Üê generate magic link ‚Üí auth browser ‚Üí save storageState
  global-teardown.ts       ‚Üê delete all [E2E]-prefixed records owned by test user
  helpers/
    admin.ts               ‚Üê Supabase admin client; seed and cleanup helpers
    constants.ts           ‚Üê shared constants (E2E_PREFIX etc.)
    routes.ts              ‚Üê typed URL builders
  tests/
    public-feed.spec.ts    ‚Üê unauthenticated user: feed, redirects
    auth.spec.ts           ‚Üê session state: nav links, redirectTo preservation
    systems.spec.ts        ‚Üê create system, edit system, add snapshot, edit snapshot
    test-creation.spec.ts  ‚Üê full wizard: track ‚Üí snapshots ‚Üí clips ‚Üí publish
    voting.spec.ts         ‚Üê cast vote, hidden/visible tally, reveal, update vote
    profile.spec.ts        ‚Üê update display name
    zz-sign-out.spec.ts    ‚Üê sign out (must run last ‚Äî see file for why)
playwright.config.ts
```

## Test coverage

### `public-feed.spec.ts` ‚Äî no auth required

| # | Scenario |
|---|---|
| 1 | Home page loads successfully |
| 2 | Header shows "Sign in" link and not authenticated nav |
| 3 | Test cards have expected structure (title, status badge) when tests exist |
| 4 | Visiting `/systems` redirects to `/login` with `redirectTo` param |
| 5 | Login page shows magic link form and Google sign-in button |
| 6 | Visiting `/profile` redirects to `/login` |
| 7 | Visiting `/tracks` redirects to `/login` |

### `auth.spec.ts` ‚Äî auth session state

| # | Scenario |
|---|---|
| 1 | Authenticated header shows Tests / Systems / Tracks / Profile nav links |
| 2 | `redirectTo` is preserved: `/systems` ‚Üí login ‚Üí lands back on `/systems` |

### `systems.spec.ts` ‚Äî system CRUD (creates and cleans up `[E2E]` data)

| # | Scenario |
|---|---|
| 1 | Create a system: fill form ‚Üí submit ‚Üí land on detail page with correct name |
| 2 | Edit the system name and description: save ‚Üí detail page shows updated values |
| 3 | Add a snapshot to the system: fill label ‚Üí submit ‚Üí snapshot appears in list |
| 4 | Edit a snapshot label: save ‚Üí updated label shown |
| 5 | Systems list shows the test user's own systems |

### `test-creation.spec.ts` ‚Äî full wizard (seeds a track and two systems)

| # | Scenario |
|---|---|
| 1 | Step 1: navigate to wizard and search for the seeded track |
| 2 | Full wizard flow: select track ‚Üí snapshots ‚Üí verify clips ‚Üí publish |

### `voting.spec.ts` ‚Äî voting flow (seeds a complete published test)

| # | Scenario |
|---|---|
| 1 | Before voting: tally is hidden but vote count is visible |
| 2 | Cast a vote: select clip A for the first technique ‚Üí Save votes |
| 3 | Update an existing vote: radios pre-filled, change to clip B |
| 4 | Creator can reveal the test |

### `profile.spec.ts`

| # | Scenario |
|---|---|
| 1 | Profile page loads and shows the display name field |
| 2 | Update display name and see confirmation |
| 3 | Save button is disabled when display name is cleared |

### `zz-sign-out.spec.ts` ‚Äî must run last

| # | Scenario |
|---|---|
| 1 | Sign out clears the session: header shows "Sign in" after |

`SignOutButton` calls `supabase.auth.signOut()` with the default **global**
scope, which revokes every session for the E2E test account ‚Äî including the
shared session in `playwright/.auth/user.json` that every other authenticated
test depends on, regardless of which browser context triggers the sign-out.
The `zz-` prefix keeps this test running last under the serial (`workers: 1`)
test order, so its collateral revocation doesn't take down the rest of the
suite.

## Environment variables

Add to `.env.local` (local runs) and to Vercel / GitHub Actions (CI):

```bash
# Required for E2E tests ‚Äî never committed
SUPABASE_SERVICE_ROLE_KEY=<staging service role key>
E2E_TEST_USER_EMAIL=e2e-tests@example.com
E2E_ADMIN_USER_EMAIL=e2e-admin@example.com  # must also be listed in ADMIN_EMAILS
E2E_BASE_URL=http://localhost:3000        # local
# E2E_BASE_URL=https://your-preview.vercel.app   # CI
VERCEL_AUTOMATION_BYPASS_SECRET=<protection bypass secret>  # only if E2E_BASE_URL is Vercel-protected
```

`NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` are already
in `.env.local` and must point to the staging project.

`VERCEL_AUTOMATION_BYPASS_SECRET` is only needed when `E2E_BASE_URL` points at
a Vercel deployment with SSO Deployment Protection enabled ‚Äî true for most
staging/preview URLs. Without it, every request gets redirected to
`vercel.com/sso-api` before ever reaching the app. Generate via
`vercel project protection enable <project> --protection-bypass`, then read
the secret back with `vercel project protection <project>`. Not needed for
`http://localhost:3000`.

## Setup: create the staging test user

The `E2E_TEST_USER_EMAIL` account must exist in the staging Supabase project
before the first test run. Create it once via the Supabase dashboard:

1. **Authentication ‚Üí Users ‚Üí Invite user**
2. Enter the E2E email address
3. Confirm the user (or use the Admin API to `createUser` with `email_confirm: true`)

The global setup will then generate magic links for this user on every run.

## Setup: create the staging admin test user (step 64)

`E2E_ADMIN_USER_EMAIL` needs the same one-time setup as `E2E_TEST_USER_EMAIL`
above, **plus** one extra step:

1. **Authentication ‚Üí Users ‚Üí Invite user** (or Admin API `createUser` with
   `email_confirm: true`), same as the regular test user.
2. Add this same email address to the staging environment's `ADMIN_EMAILS`
   Vercel env var (comma-separated) ‚Äî `isAdminEmail()` reads it at request
   time, so without this step the account signs in fine but never passes
   the admin gate.

Keep this a distinct address from `E2E_TEST_USER_EMAIL` ‚Äî do not add the
regular test user to `ADMIN_EMAILS` instead of creating a second account.
Several specs (`profile.spec.ts`, `admin-clip-override.spec.ts`) assert that
the *regular* test user is **not** an admin; making it one would silently
break those negative-case assertions.

## Running the tests

```bash
# Start the dev server first (or set E2E_BASE_URL to a Vercel preview URL)
npm run dev

# Run all E2E tests
npm run test:e2e

# Run with Playwright's interactive UI
npm run test:e2e:ui

# Run with debug inspector
npm run test:e2e:debug

# Run a specific spec
npx playwright test e2e/tests/systems.spec.ts
```

## CI integration

E2E tests should **not** run on every push ‚Äî they are slower and write to the
staging database. Recommended triggers:

- **On merge to `staging` branch** ‚Äî verifies the staging Vercel deployment is
  healthy; `E2E_BASE_URL` set to the staging Vercel URL
- **On demand** ‚Äî `workflow_dispatch` in GitHub Actions before merging to `main`

Do not run E2E on feature branches or PRs ‚Äî concurrent runs against the same
staging database can conflict on test data even with the `[E2E]` prefix.

---

## Teardown: deletion order

The schema has no `ON DELETE CASCADE` constraints, so teardown deletes in
dependency order:

```
votes            (references tests)
clip_mapping     (references tests + clips)
clips            (references tests)
tests            (references tracks + system_snapshots)
system_snapshots (references systems)
systems
tracks
```

All deletions are scoped to `[E2E]`-prefixed records owned by the test user.
